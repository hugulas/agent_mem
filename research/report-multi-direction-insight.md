## 方向一：记忆存储层——文件的复兴 vs 向量的霸权

### 1.1 四种存储哲学

2026 年的 Agent 记忆存储层正在经历一场静默的革命：**纯文本文件（Markdown）作为存储介质重新回到了中心舞台**，而 2024 年被视为"标配"的向量数据库正被重新评估其必要性。

| 存储哲学 | 代表系统 | 核心形态 | 典型规模 | 优势 | 劣势 |
|---|---|---|---|---|---|
| **文件即真相** | OpenClaw, Claude Code, Codex, Hermes, Karpathy Wiki | Markdown + 可选索引 | KB–MB 级 | 人类可读、Git 可版本化、零 vendor lock-in | 缺乏结构化查询能力、大文件扫描慢 |
| **向量数据库** | 传统 RAG, Mem0, Pinecone | Embedding 向量 + 元数据 | GB–TB 级 | 语义检索、规模弹性、成熟生态 | 黑盒、embedding 漂移、切片噪声 |
| **文件系统范式** | OpenViking | `viking://` 虚拟目录 + L0/L1/L2 | MB–GB 级 | 结构化导航 + 语义检索兼得 | 需要外部 LLM 生成摘要、setup 复杂 |
| **模型参数内** | UI-TARS, Gemini | Transformer KV Cache / 模型权重 | GB 级（单用户 15–30GB） | 无外部依赖、检索即推理 | 不可控、不可审计、成本极高 |
| **设备端向量索引** | Apple Intelligence, 豆包手机 | 本地 embedding + 向量 DB | 10–100 MB | 敏感数据不出设备、低延迟检索 | 受限于设备算力和存储 |

### 1.2 关键洞察

**洞察 1.1：Markdown 的复兴不是倒退，而是对"可审计性"的回归**

所有主流开源框架（OpenClaw、Hermes、Claude Code、Codex）以及 Karpathy Wiki 都以 Markdown 为存储底座。这并非因为开发者缺乏数据库技能，而是因为：
- **可审计性**：任何claim都可追溯到具体文件的具体行。
- **可干预性**：用户可以直接用文本编辑器修改记忆，无需 API。
- **版本化**：Git 天然支持记忆的 diff、branch 和 rollback。
- **LLM 友好**：LLM 对 Markdown 语法（标题、列表、代码块、链接）的理解远优于对数据库 schema 的理解。

**洞察 1.2：向量数据库正在从"主角"退化为"配角"**

2024 年的共识是"Agent 记忆 = 向量数据库"。2026 年的现实是：
- OpenClaw 的向量搜索是 SQLite 的**派生索引**，原始真相源是 Markdown 文件。
- Claude Code 和 Codex CLI **完全不使用向量**——前者用 LLM 扫描文件头，后者用 grep。
- Hermes 的 FTS5 全文检索（113 ms）在多数场景下比向量检索（19,593 ms）更快。
- Karpathy 直接用 LLM 读取整个 Wiki，宣称在 400K 词规模下"不需要 RAG"。

向量数据库并未消失，但其角色从"唯一存储"降级为"可选加速器"。

**洞察 1.3：中国终端厂商的"系统级记忆"走了一条独特路线**

与开源框架的"文件优先"不同，中国手机厂商的记忆存储更偏向**系统级服务化**：
- **vivo OriginOS 小V记忆 2.0**：跨平台收藏 + AI 提炼分类 + 关联日程/地址，存储于系统级云服务。
- **小米 miclaw**：三级智能记忆管理（自动保留关键决策点 / 动态压缩冗余 / 核心指令本地缓存），端云隐私计算。
- **OPPO ColorOS 16 小布记忆**：PersonaX 记忆共生引擎，系统级多模态、全时全域感知、端侧优先隐私计算。
- **Kimi 上下文缓存**：Context Caching 降本 90%，将长上下文复用从存储层优化。

这些系统的共同特点是**深度集成操作系统**，记忆存储不是独立文件或数据库，而是系统服务的一部分——用户无法直接编辑，但获得了跨应用的无缝体验。

## 方向二：检索机制——从相似度匹配到认知导航

### 2.1 检索机制光谱

| 检索机制 | 代表系统 | 原理 | 延迟 | 精确性 | 可解释性 |
|---|---|---|---|---|---|
| **混合语义+关键词** | OpenClaw | 向量相似度 70% + BM25 30% + MMR + 时间衰减 | ~19.6s（300 事件） | 中（语义灵活） | 低（embedding 黑盒） |
| **全文检索（FTS5）** | Hermes | SQLite 全文索引 + BM25 排名 | ~10–113 ms | 高（确定性） | 高（查询可重现） |
| **LLM 文件头扫描** | Claude Code | 模型逐层读取 CLAUDE.md 各级标题 | ~数百 ms–数 s | 高（理解深度） | 中（消耗 API tokens） |
| **确定性 grep** | Codex CLI | 全文加载 summary + grep 细节 | ~毫秒级 | 中（无语义变体） | 极高（完全透明） |
| **目录递归+语义** | OpenViking | 目录定位 → 子目录二次搜索 → 逐级下钻 | ~数百 ms | 高（路径约束） | 极高（可视化轨迹） |
| **注意力机制全上下文** | Gemini 2M | 模型直接 attend 到 2M token 内的任意位置 | ~10–30s（预填充） | 高（无检索损失） | 极低（模型黑盒） |
| **模型内情景召回** | UI-TARS | Et 语义压缩摘要参与当前推理 | 模型推理延迟 | 中（依赖压缩质量） | 极低（不可干预） |
| **语义索引+App Intents** | Apple Intelligence | 设备端向量数据库 + 应用意图匹配 | ~100 ms | 高（个人上下文） | 低（系统黑盒） |

### 2.2 关键洞察

**洞察 2.1：检索延迟差距可达 3 个数量级**

同一 300 事件基准下：Hermes FTS5 113 ms vs OpenClaw 混合搜索 19,593 ms——差距 **173 倍**。这不是技术代差，而是架构哲学的差异：
- Hermes 用数据库查询（确定性、预计算索引）。
- OpenClaw 将完整 JSONL 历史注入 LLM 上下文（非确定性、每轮重新推理）。

对于生产环境，这种延迟差异意味着：需要频繁检索记忆的场景（如多通道聊天机器人）更适合 FTS5/目录递归；需要深度语义理解的场景（如研究分析）才需要向量/LLM 扫描。

**洞察 2.2：OpenViking 的"目录递归"是一种被低估的中间路线**

传统 RAG 的问题是"扁平化"——所有文档被切成等权重的向量块，丢失了原始的结构化关系（章节、目录、层级）。OpenViking 的创新在于**保留并利用这种结构**：
- 先定位到正确目录（粗筛，低成本）。
- 再在目录取内语义搜索（精筛，高相关性）。
- 检索轨迹可视化使调试成为可能。

这在认知上更接近人类查找资料的方式：先看目录 → 再翻章节 → 最后读段落。

**洞察 2.3：中国厂商的检索更倾向于"场景化关联"而非"查询式检索"**

- vivo 小V记忆：收藏内容自动关联日程、地址、天气——检索由**场景触发**（如到达某地点时自动推送相关收藏），而非用户主动查询。
- OPPO PersonaX：全时全域感知意味着检索是**持续被动**的，系统随时根据上下文预加载相关记忆。
- 小米 miclaw：三级记忆中，检索优先级由"关键决策点 > 动态压缩 > 本地缓存"的层级决定。

这种"主动推送"模式与开源框架的"被动查询"模式形成了鲜明对比。

## 方向三：上下文压缩——空间焦虑的七种缓解方案

### 3.1 压缩策略全景

所有 Agent 框架都面临同一个根本约束：**上下文窗口有限，而记忆无限**。2026 年的解决方案可分为七大类：

| 策略 | 代表系统 | 核心机制 | 信息损失 | 适用场景 |
|---|---|---|---|---|
| **渐进式压缩管道** | Claude Code | 预算→截断→微压缩→折叠→自动压缩（5层） | 分层递增 | 编码场景，工具结果频繁 |
| **分层按需加载** | OpenViking | L0 摘要 → L1 概览 → L2 全文 | 按需零损失 | 文档密集型 Agent |
| **超长上下文消化** | Gemini 2.5 Pro | 2M token 原生窗口 + KV Cache | 无（全量保留） | 一次性分析大量文档 |
| **单层 handoff** | Codex CLI | 会话结束时生成单页 summary | 较高（仅保留 summary） | 快速编码，极简运维 |
| **预压缩记忆刷新** | OpenClaw | 压缩前自动 flush 记忆到文件 | 中（flush 后压缩） | 多通道长期运行 Agent |
| **模型内语义压缩** | UI-TARS | Et 将历史 episode 压缩为语义摘要 | 中（依赖压缩质量） | 原生 Agent 模型 |
| **端侧分层+云端大模型** | 豆包/Apple | 本地缓存轻量上下文，复杂推理上云 | 低（云端全量） | 手机/设备端 Agent |

### 3.2 关键洞察

**洞察 3.1：压缩是"必要之恶"，但不同场景对损失的容忍度差异巨大**

- **编码场景**（Claude Code）：允许较高损失，因为代码可以通过重新执行验证。Claude 的 5 层压缩管道中最激进的"上下文折叠"会丢弃大量中间状态，但保留关键决策点。
- **医疗/法律场景**（理论上）：几乎不允许损失，需要 OpenViking 式 L0/L1/L2 的按需零损失加载，或 Gemini 2M 的全量保留。
- **个人助手场景**（OpenClaw/Hermes）：中等容忍度，但需要可审计——用户需要知道"什么被保留了、什么被丢弃了"。

**洞察 3.2：超长上下文正在改变压缩的必要性**

Gemini 2M（以及 Claude 1M、Llama 10M）的上下文窗口使得"压缩"在很多场景下变得不必要。一个 400K 词的 Wiki（Karpathy 的规模）可以完全放入 Gemini 的上下文窗口，无需任何 RAG 或压缩。

但这带来了新的成本结构：
- **时间成本**：2M token 预填充延迟 2 分钟+。
- **内存成本**：每用户 15–30 GB KV Cache。
- **金钱成本**：长上下文 API 调用费用数倍于短上下文。

因此，压缩并未消失，而是从"必需"变为"成本优化手段"。

**洞察 3.3：中国手机厂商的压缩策略更偏向"系统级智能调度"**

- 小米 miclaw 的"动态压缩冗余"不是 Agent 框架内的功能，而是**系统级资源管理**——在 OS 层面判断哪些 Agent 状态需要保留、哪些可以丢弃。
- Kimi 的 Context Caching 将压缩/复用从应用层下沉到**模型服务层**，对应用开发者透明。

这意味着终端厂商正在将"上下文压缩"从 Agent 框架的差异化功能，转变为**操作系统的标准化能力**。

## 方向四：部署形态——本地、云端与端云的权力分配

### 4.1 五种部署形态

| 形态 | 代表系统 | 数据主权 | 延迟 | 运维负担 | 规模化 |
|---|---|---|---|---|---|
| **完全本地自托管** | OpenClaw, Hermes, Claude Code | 用户完全拥有 | 低（无网络） | 高（需自行维护） | 差（单机限制） |
| **个人设备内置** | Apple Intelligence, 豆包手机 | 设备厂商 + 用户 | 极低（端侧） | 无（系统更新） | 差（换机迁移） |
| **端云协同** | 豆包手机（云端推理+本地检索） | 分割（敏感本地、复杂云端） | 中（~250KB/3-5s 上传） | 低 | 中 |
| **云端托管** | Copilot, Bedrock, Astra | 云厂商 | 中–高（受网络影响） | 极低 | 极佳 |
| **模型即服务** | UI-TARS, Gemini API | 模型提供商 | 高（API 往返） | 无 | 极佳 |

### 4.2 关键洞察

**洞察 4.1：部署形态决定了记忆系统的"权力结构"**

- **完全本地**（OpenClaw/Hermes）：用户拥有绝对控制权，但也承担全部责任（备份、安全、更新）。
- **云端托管**（Copilot/Bedrock）：用户放弃数据主权，换取便利性和跨设备同步。记忆可以被企业审计（Purview），也可能被厂商用于模型改进（取决于隐私政策）。
- **端云协同**（豆包/Apple）：试图在两者之间取得平衡——敏感记忆留在本地，复杂推理上云。但这也创造了**新的攻击面**：本地↔云端的数据同步通道。

**洞察 4.2：中国终端厂商的"系统级集成"是一种无法被开源框架复制的部署优势**

开源框架（OpenClaw/Hermes）运行在用户空间，受限于操作系统的沙箱。而 vivo/小米/OPPO 的记忆系统：
- 可以读取**任何应用的屏幕内容**（系统级辅助功能权限）。
- 可以**跨应用保持状态**（不受应用生命周期限制）。
- 可以**利用系统级推送**在合适时机触发记忆检索。

这解释了为什么终端厂商的 Agent 记忆体验往往比第三方框架更"无缝"——不是因为技术更先进，而是因为**部署位置更接近系统核心**。

## 方向五：隐私与治理——信任锚点的五种设计哲学

### 5.1 隐私光谱

| 设计哲学 | 代表系统 | 信任锚点 | 用户控制 | 企业治理 | 透明度 |
|---|---|---|---|---|---|
| **本地完全可控** | OpenClaw, Hermes, Karpathy Wiki | 用户自己的文件系统 | 极高（直接编辑文件） | 无 | 完全透明 |
| **密码学隐私保证** | Apple PCC | 硬件+密码学证明 | 低（系统黑盒） | 无 | 中等（可验证但不可读） |
| **企业合规治理** | Microsoft Copilot | 租户管理员 + Purview 审计 | 中（用户可查看/删除，管理员可审计） | 强（RBAC/DLP） | 中（审计日志） |
| **云厂商信任** | Amazon Bedrock, Google Astra | AWS/Google 安全团队 | 低（依赖厂商 UI） | 依赖厂商 | 低（黑盒） |
| **模型黑盒** | UI-TARS | 模型提供商 | 极低（不可干预） | 无 | 无 |

### 5.2 关键洞察

**洞察 5.1：没有"绝对隐私"，只有"信任转移"**

- 开源框架将信任从云厂商转移到了**用户自身**——但用户需要自行承担安全责任（CVE-2026-25253 显示 OpenClaw 的明文存储存在严重风险）。
- Apple PCC 将信任从 Apple 公司转移到了**密码学硬件**——零数据保留、密码学可验证，但用户仍然需要信任 Apple 的硬件实现没有后门。
- Microsoft Copilot 将信任转移到了**企业 IT 部门**——用户信任的不是 Microsoft，而是自己公司的安全策略。

**洞察 5.2：中国终端厂商的隐私策略强调"端侧优先"**

- 小米 miclaw 的"端云隐私计算"将敏感记忆加密后在本地处理。
- OPPO PersonaX 明确将"端侧优先隐私计算"作为核心卖点。
- 这与 Apple 的隐私优先策略异曲同工，但实现方式不同：Apple 依赖 PCC 密码学证明，中国厂商更多依赖**TEE（可信执行环境）+ 联邦学习**的组合。

**洞察 5.3：记忆投毒是所有架构的共同威胁**

无论部署形态如何，持久化记忆都面临**记忆投毒（Memory Poisoning）**攻击：
- Trojan Hippo 研究显示：对 Gemini-3.1-pro 的记忆投毒成功率可达 **100%**（无防御时），对 GPT-5-mini 达 **85%**。
- 开源框架中，OpenClaw 的恶意 Skill 可向向量数据库写入持久化后门规则。
- 云厂商中，Copilot 的 intent-driven 存储虽然减少了"意外记忆"，但也增加了"被诱导记忆"的风险。

## 方向六：可观测性——从黑盒到白盒的记忆调试

### 6.1 可观测性层级

| 层级 | 代表系统 | 可观测对象 | 调试能力 |
|---|---|---|---|
| **完全黑盒** | Gemini 2M, UI-TARS | 无（模型内部注意力权重不可读） | 无 |
| **检索日志** | OpenClaw, 传统 RAG | 搜索查询、返回结果、相似度分数 | 有限（知道"检索了什么"，但不知道"为什么"） |
| **文件级透明** | Claude Code, Codex, Karpathy Wiki | 完整的记忆文件内容 | 高（可直接阅读、编辑、版本对比） |
| **可视化轨迹** | OpenViking | 目录遍历路径、每一步的决策理由 | 极高（可回放检索过程，定位错误节点） |
| **审计日志** | Microsoft Copilot | 记忆创建/修改/删除的日志 | 高（企业级审计） |

### 6.2 关键洞察

**洞察 6.1：Agent 失败的 80% 不是模型失败，而是"上下文路由失败"**

OpenViking 的团队提出一个关键观点：很多 Agent 失败并非因为模型能力不足，而是因为**检索到了错误的上下文**。如果 Agent 在 50 份运维手册中检索到了不相关的一份，即使 GPT-5 也会给出错误答案。

这意味着**检索可观测性**与模型能力同等重要——甚至更重要。

**洞察 6.2：Karpathy Wiki 的"编译器模式"天然具备最高可观测性**

因为 Wiki 是结构化 Markdown，开发者可以：
- 直接阅读任何一篇概念文章，验证其准确性。
- 查看 backlinks，理解知识之间的连接。
- 运行"linting"健康检查，发现矛盾和不一致。

这种可观测性在向量数据库中是不可能的——你无法"阅读"一个 embedding 向量。

## 方向七：记忆生命周期——谁负责记忆的生老病死

### 7.1 记忆生命周期对比

| 生命周期阶段 | 开源框架 | OpenViking | 云厂商 | 终端厂商 | Karpathy Wiki |
|---|---|---|---|---|---|
| **生成（Formation）** | 用户手动写 / Agent 自动提取 | LLM 自动分层（L0/L1/L2） | 自动提取（意图驱动） | 系统级自动感知 | LLM 编译 raw 资料 |
| **存储（Storage）** | Markdown / SQLite / 向量 DB | viking:// 虚拟文件系统 | 云端托管数据库 | 设备端 + 云端混合 | Markdown 文件 |
| **检索（Retrieval）** | 搜索 / grep / LLM 扫描 | 目录递归 + 语义 | 语义相似度 + 作用域过滤 | 场景化推送 + 查询 | 直接读取相关文件 |
| **更新（Update）** | 用户编辑 / 后台合并 | 自动会话提取 + 自迭代 | 动态更新（用户可编辑） | 系统级同步 | LLM linting + 增量编译 |
| **遗忘（Forgetting）** | 时间衰减 / 手动删除 | 未明确（新兴需求） | 自动保留策略 | 系统资源管理 | 未明确（文件持续增长） |

### 7.2 关键洞察

**洞察 7.1："遗忘"是所有记忆系统中最被忽视的环节**

人类大脑通过睡眠巩固和遗忘来维持记忆的相关性。但当前 Agent 记忆系统几乎都没有成熟的遗忘机制：
- OpenClaw 有时间衰减（30 天 half-life），但这是简单的权重调整，不是真正的删除。
- Codex 有 30 天 age-out，但这是基于时间的硬删除，不考虑信息重要性。
- 云厂商（Copilot/Bedrock）的保留策略由管理员设定，同样不考虑语义相关性。
- **没有任何主流系统实现了基于"记忆冲突检测"的智能遗忘**——例如，当用户说"我不再吃海鲜了"，系统应该主动弱化/删除之前关于"用户喜欢海鲜"的记忆。

这是一个巨大的产品空白，也是下一代记忆系统的核心差异化方向。

**洞察 7.2：Karpathy Wiki 的"知识复利"需要新的生命周期管理**

Wiki 模式的问题是**只增不减**——每次查询的输出被归档回 Wiki，使知识库不断增长。虽然 Karpathy 说 400K 词仍在舒适区，但长期运行后：
- 旧知识可能过时（技术文档、API 变更）。
- 新知识可能与旧知识矛盾。
- 文件数量膨胀导致检索效率下降。

目前 Karpathy 用"linting"健康检查来发现不一致，但没有自动的"知识退役"机制。这是编译器模式需要解决的下一个问题。

## 方向八：生态位与商业模式——谁在卖铲子，谁在卖金子

### 8.1 市场地图

| 生态位 | 玩家 | 卖什么 | 商业模式 | 护城河 |
|---|---|---|---|---|
| **开源框架** | OpenClaw, Hermes | 可定制性 + 社区 | 免费（捐赠/企业支持） | 社区规模、插件生态 |
| **上下文数据库** | OpenViking, Mem0 | 检索效率 + 可观测性 | 开源 + 云服务 | 技术壁垒、集成广度 |
| **模型内置** | UI-TARS, Gemini | 推理能力 + 上下文长度 | API 调用费 | 模型质量、算力规模 |
| **云托管服务** | Copilot, Bedrock | 零运维 + 企业治理 | 订阅制 / 按调用 | 企业客户锁定、合规认证 |
| **设备系统** | Apple, vivo, 小米, OPPO | 隐私 + 无缝体验 | 硬件销售 | 系统级权限、用户粘性 |
| **工作流范式** | Karpathy Wiki | 理念 + 最佳实践 | 免费（影响力） | 思想领导力、社区传播 |

### 8.2 关键洞察

**洞察 8.1：记忆系统正在分层为"基础设施层"和"体验层"**

- **基础设施层**（OpenViking、Bedrock AgentCore、S3 Vectors）：卖的是检索效率、存储成本、可扩展性。客户是开发者和企业。
- **体验层**（Copilot、Apple Intelligence、豆包手机）：卖的是用户感知到的"智能"和"贴心"。客户是终端消费者。
- **中间层正在消失**：2024 年有很多独立的"记忆即服务"初创公司（如早期 Mem0），但 2026 年它们要么被框架集成（Mem0 → Hermes 提供商），要么被云厂商收编（向量存储 → S3 Vectors）。

**洞察 8.2：终端厂商可能是记忆系统的"终极赢家"**

因为记忆的价值随**数据量**和**场景覆盖度**递增，而终端厂商（Apple、vivo、小米、OPPO）拥有：
- 最大的数据入口（屏幕、传感器、应用使用）。
- 最深的系统集成（不受应用沙箱限制）。
- 最强的用户粘性（换机成本）。

开源框架和云厂商虽然技术更开放，但在"谁更了解用户"这个维度上，永远追不上每天与用户 24 小时相处的手机系统。

## 跨方向综合洞察与选型矩阵

### 综合洞察

1. **记忆系统没有"最佳"架构，只有"最适配场景"的架构**。编码场景需要确定性和低延迟（Claude Code/Hermes）；研究场景需要知识复利和可审计性（Karpathy Wiki）；企业场景需要治理和零运维（Copilot/Bedrock）；设备场景需要隐私和无缝（Apple Intelligence/豆包）。

2. **2026 年的核心矛盾是"可扩展性" vs "可审计性"**。向量数据库和超长上下文在扩展性上获胜，但黑盒不可审计；Markdown 文件和可视化轨迹在可审计性上获胜，但在大规模下效率受限。OpenViking 是目前唯一试图同时解决这两个问题的系统。

3. **中国厂商与美国厂商走在了不同的道路上**。美国厂商（Google/Microsoft/Amazon）倾向于用**云和模型**解决记忆问题；中国终端厂商（vivo/小米/OPPO）倾向于用**系统和设备**解决记忆问题；开源社区（OpenClaw/Hermes/Karpathy）倾向于用**文件和协议**解决记忆问题。三条路线各有优劣，短期内不会收敛。

4. **"遗忘"是下一个战场**。当前所有系统在记忆的生命周期管理上都存在明显短板——尤其是智能遗忘和冲突解决。谁先解决这个问题，谁就可能在个性化 Agent 体验上取得决定性优势。

### 多维度选型矩阵

| 你的需求 | 推荐方案 | 关键理由 |
|---|---|---|
| **需要完全控制记忆内容** | OpenClaw / Hermes / Karpathy Wiki | Markdown 文件可直接编辑、Git 版本化 |
| **需要最低检索延迟** | Hermes (FTS5) / OpenViking | 113 ms vs 19,593 ms 的实测差距 |
| **需要处理 1000+ 份文档** | OpenViking / 传统 RAG | L0/L1/L2 分层或向量数据库才能扩展 |
| **需要企业合规审计** | Microsoft Copilot / Azure Foundry | Purview + Entra Agent ID + RBAC |
| **需要跨设备无缝同步** | Apple Intelligence / 云厂商方案 | 系统级集成或账户级同步 |
| **需要最高隐私保证** | Apple PCC / 完全本地开源框架 | 密码学证明或物理隔离 |
| **需要知识复利（研究/学习）** | Karpathy Wiki + Claude Code | 编译器模式，零基础设施 |
| **需要 GUI 自动化记忆** | 字节 UI-TARS / 豆包手机 | 原生模型内记忆 + 系统级屏幕理解 |
| **需要最低运维成本** | Amazon Bedrock AgentCore | 完全托管，50 行代码接入 |
| **需要可视化调试检索** | OpenViking | 唯一提供检索轨迹可视化的系统 |
