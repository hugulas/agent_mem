# Hermes issue #23367: Context compression integration

> GitHub issue | 2026-05-10

## 1. Core Claim
- Hermes' existing memory provider compression hook (`on_pre_compress`) is too limited — it only returns unstructured text, preventing providers from specifying preservation directives or receiving feedback on what the compressor kept or discarded.

## 2. Method / Mechanism
- **Current behavior**: `on_pre_compress(messages)` receives full messages before pruning; providers can append freeform text to the compression prompt.
- **Limitations identified**: (1) No structured return type, (2) No preservation directives, (3) No feedback loop, (4) Text is appended but not structurally preserved in the output summary.
- **Proposed improvement**: `CompressionContext` dataclass with fields:
  - `text`: freeform context (backward-compatible)
  - `preserve_snippets`: text that MUST appear in summary
  - `preservation_rules`: declarative rules (e.g., "keep_last_file_path")
  - `key_facts`: facts to inject verbatim into final summary
- **Use case**: Muninn Memory Plugin could preserve file paths, URLs, decisions, and corrections across compression.

## 3. Key Evidence
- Issue includes concrete pseudocode for `CompressionContext`, `_generate_summary()`, and a `PRESERVATION_RULES` engine.
- Demonstrates awareness that current integration loses critical information during context compression.
