# OpenViking GitHub repo (volcengine/OpenViking)

> Official repo | 2026-05

## 1. Core Claim
- Agent memory should be managed through a filesystem paradigm rather than fragmented vector databases, enabling unified organization of memories, resources, and skills with tiered, on-demand context loading.

## 2. Method / Mechanism
- **Filesystem Paradigm (viking://)**: Maps memories, resources, and skills into a virtual hierarchical filesystem with directories like `user/`, `agent/`, and `resources/`.
- **Tiered Context Loading (L0/L1/L2)**:
  - L0: ~100-token abstract/summary for quick identification.
  - L1: ~2K-token overview for planning.
  - L2: Full original content for deep reading when necessary.
- **Directory Recursive Retrieval**: Vector retrieval first identifies a high-score directory, then performs secondary retrieval within that directory, recursively drilling into subdirectories.
- **Visualized Retrieval Trajectory**: Stores and exposes the path of directory browsing/file positioning during retrieval for debugging.
- **Automatic Session Management**: Auto-compresses conversation content, resource references, and tool calls to extract long-term memory.
- **Tech Stack**: Rust (core) + Python SDK + CLI (npm); supports Ollama, OpenAI, Volcengine, Gemini, and 10+ embedding providers.

## 3. Key Evidence
- 24k+ GitHub stars; #1 trending repository at time of capture.
- 155 contributors; active development (commits within hours).
- Apache 2.0 license (previously AGPLv3 noted in some metadata).
- Supports Vaka memory templates and OpenClaw skill integration (`ov_dream`).

## 4. Limitations & Gaps
- Newer project (v0.3.17); production maturity still evolving.
- Tiered loading requires upfront annotation/generation of L0/L1 summaries — overhead on write path.
- Retrieval trajectory visualization helps debugging but adds storage/complexity.
- Multi-provider support is broad but may have inconsistent behavior across backends.
