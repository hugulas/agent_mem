# RobotPaper: Reference Architecture OpenClaw

> Technical analysis | 2026-02-24

## 1. Core Claim
OpenClaw implements a **layered agent runtime** with a Gateway daemon as the central nervous system, an embedded agent runtime (pi-mono) for LLM interaction, and a four-layer memory stack that makes a stateless LLM behave like a stateful, continuously-available assistant.

## 2. Method / Mechanism
- **Gateway (Node.js daemon)**: Single long-lived process owning all state and connections. WebSocket-first protocol on port 127.0.0.1:18789. Device pairing with trust model (auto-approve local, challenge nonce for remote).
- **Channel Bridges**: WhatsApp (Baileys), Telegram (grammY), Discord, Slack, Signal, iMessage, WebChat.
- **Agent Runtime (pi-mono)**: Prompt assembly, tool execution, compaction pipeline, memory search, streaming engine, sub-agent spawner, skill loader, sandbox manager.
- **Four-layer memory stack**:
  - Layer 4: Semantic vector search (SQLite + embeddings)
  - Layer 3: Long-term Memory (MEMORY.md) — curated insights
  - Layer 2: Daily Logs (memory/YYYY-MM-DD) — append-only raw notes
  - Layer 1: Commitments / ephemeral follow-ups
- **Queue system**: Lane-aware FIFO (global lane maxConcurrent:4, session lane concurrency:1, sub-agent lane:8).
- **Chunking**: ~400 tokens per chunk / 80 token overlap.
- **Hybrid search**: 70% vector + 30% BM25.
- **File watcher**: Debounced at 1.5s for live index updates.

## 3. Key Evidence
- Chunking parameters: ~400 tokens / 80 overlap.
- Hybrid search weights: 70% vector + 30% BM25.
- Local GGUF embedding model: ~0.6GB.
- File watcher debounce: 1.5s.
- Gateway architecture diagram showing clear separation between messaging surfaces, gateway, agent runtime, and LLM providers.

## 4. Limitations & Gaps
- This is a **community-generated architecture doc** (Roy Osherove asked OpenClaw with Opus 4.6 to analyze itself), not official documentation — potential hallucination risk in architectural claims.
- No quantitative benchmarks on memory recall accuracy or latency.
- AWS-focused deployment discussion may not generalize to other cloud providers.
- The author notes they did not use Claude Code, so comparisons may be one-sided.

## 5. Relevance to Our Review
- **OpenClaw architecture** — most detailed third-party architectural walkthrough available
- **Memory stack design** — four-layer hierarchy as a representative file-first approach
- **Gateway pattern** — contrast with Claude Code's single CLI loop and Hermes's multi-channel approach
