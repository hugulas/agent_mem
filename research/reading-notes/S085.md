> How It Works
> When a memory provider is active, Hermes automatically:
> 1. Injects provider context into the system prompt (what the provider knows)
> 2. Prefetches relevant memories before each turn (background, non-blocking)
> 3. Syncs conversation turns to the provider after each response
> 4. Extracts memories on session end (for providers that support it)
> 5. Mirrors built-in memory writes to the external provider
> 6. Adds provider-specific tools so the agent can search, store, and manage memories

## 1. Core Claim
- Hermes Agent ships with 8 external memory providers under a unified integration model; the built-in MEMORY.md/USER.md memory is always active, and exactly one external provider can be added at a time.

## 2. Method / Mechanism
- **Unified provider lifecycle**: On activation, Hermes automatically (1) injects provider context into system prompt, (2) prefetches memories before each turn, (3) syncs conversation turns, (4) extracts memories on session end, (5) mirrors built-in writes, and (6) registers provider-specific tools.
- **Holographic-specific capabilities**: 9 `fact_store` actions, `fact_feedback` for trust-score training, `probe`, `reason`, and `contradict` unique capabilities.
- **Configuration**: Per-provider config files (e.g., `$HERMES_HOME/holographic.json`); interactive `hermes memory setup` wizard.

## 3. Key Evidence
- 8 providers documented: Honcho, OpenViking, Mem0, Hindsight, Holographic, RetainDB, ByteRover, Supermemory.
- Holographic config defaults: `auto_extract: false`, `default_trust: 0.5`, `min_trust_threshold: 0.3`.
- Multi-peer observation modes (directional vs. unified) for Honcho integration.

## 4. Limitations & Gaps
- Only one external provider can be active at a time — no compositional memory stacking.
- Provider quality varies widely; docs do not offer quantitative comparison.
- Some providers (e.g., RetainDB) are proprietary/cloud-only, creating vendor-lock-in risks.
