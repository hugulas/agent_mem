> How It Works
> When a memory provider is active, Hermes automatically:
> 1. Injects provider context into the system prompt (what the provider knows)
> 2. Prefetches relevant memories before each turn (background, non-blocking)
> 3. Syncs conversation turns to the provider after each response
> 4. Extracts memories on session end (for providers that support it)
> 5. Mirrors built-in memory writes to the external provider
> 6. Adds provider-specific tools so the agent can search, store, and manage memories

## 1. Core Claim
- Hermes integrates ByteRover via pre-compression extraction hooks, a local knowledge tree, and three provider-specific tools, enabling agent-native memory curation within the Hermes agent framework.

## 2. Method / Mechanism
- **Pre-compression extraction hook**: ByteRover extracts structured memories before Hermes' context compressor runs, preserving knowledge that might otherwise be lost in summarization.
- **Knowledge tree storage**: ByteRover data lives at `$HERMES_HOME/byterover/` as a local filesystem hierarchy.
- **Provider tools**: `brv_query` (search), `brv_curate` (organize/edit), `brv_status` (inspect tree state).
- **Cloud sync**: SOC2 Type II certified cloud sync available for cross-device persistence.
- Follows Hermes' standard 6-step provider activation model.

## 3. Key Evidence
- Integration uses the same unified provider framework as Honcho, Mem0, Hindsight, etc.
- SOC2 Type II certification is noted for the cloud sync tier — rare among open-source memory tools.
- Pre-compression hook is a specific technical detail that addresses a known gap in memory provider integration (see S093).

## 4. Limitations & Gaps
- Cloud sync costs $19/mo Pro; team tier is $35/user/mo — pricing may limit adoption.
- SOC2 mention implies enterprise readiness, but limited details on scope or audit date.
