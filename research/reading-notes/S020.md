## 1. Core Claim
Hermes Agent can be **deployed on Railway's serverless infrastructure** with under 512 MB RAM at idle, using a persistent volume at `/data` for skills, memory, and configuration, making it suitable for low-cost continuous operation through messaging channels.

## 2. Method / Mechanism
- **Railway template**: Deploys Hermes Agent with Python-based web management layer and Basic Auth-protected config UI.
- **Idle resource usage**: Under 512 MB RAM at idle.
- **Persistent volume**: Mounted at `/data` — preserves config, workspace, sessions, skills, and memory across redeploys.
- **Multi-provider support**: OpenRouter, DeepSeek, GLM/Z.AI, Kimi, MiniMax — no vendor lock-in.
- **Messaging channels**: Telegram, Discord, Slack, WhatsApp, Signal, CLI.
- **Gateway**: Launches automatically as managed subprocess if provider API key configured.
- **Environment variables**: PORT (8080), ADMIN_USERNAME, ADMIN_PASSWORD.
- **Dependencies**: Python 3.11+, Node.js (installed during build).

## 3. Key Evidence
- Under 512 MB RAM at idle.
- Persistent volume at `/data` for state.
- 362 total projects using the template, 217 active, 100% success on recent deploys.
- Supports 6 messaging platforms + CLI.

## 4. Limitations & Gaps
- No information on memory usage under active load (only idle specified).
- No disk space requirements for skills, memory, or SQLite databases.
- Railway-specific — may not translate to other platforms (AWS, GCP, Azure).
- No discussion of backup/restore for the persistent volume.
- Web UI requires public endpoint during setup; security implications if left exposed.
