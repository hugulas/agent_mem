> The Memory capability (#30) focuses on content memory — facts, observations, and task state. But

> These are fundamentally different:

> User modeling: "This user is a senior engineer who prefers terse responses and hates when I
> explain obvious things" → behavioral adaptation
> How Hermes solves this
> Hermes integrates with Honcho, an external user modeling service, as a separate tier from its curated
> memory:
> Three tools:
> 1. honcho_profile : Retrieves a structured peer card:
> 2. honcho_search : Semantic search over stored context — not just keyword matching, but meaning-
> based retrieval across all past interactions.
> 3. honcho_context : Dialectic Q&A — the service synthesizes answers from interaction history using
> an LLM, answering questions like "What does Alice think about microservices?" by reasoning over
> past conversations.
> Integration pattern:
> Session key mapping: Conversations are mapped to Honcho sessions via
> build_session_key(platform, user_id, thread_id)
> Prefetch strategy: Background thread prefetches user context during the previous turn (see 
> Speculative Prefetch for Capabilities (async background data loading) #107)

> Non-blocking: If prefetch isn't ready, the turn proceeds without user context

## 1. Core Claim
- User modeling (behavioral adaptation) should be treated as a distinct capability from content memory, requiring a dedicated protocol with structured profiles, semantic search, and dialectic Q&A — as demonstrated by Honcho's integration with Hermes.

## 2. Method / Mechanism
- **UserModelingBackend Protocol**: Abstract protocol with methods:
  - `get_profile()` — structured peer card (name, role, preferences, communication style, expertise, current context).
  - `search()` — semantic search over stored interaction history.
  - `ask()` — dialectic Q&A synthesizing answers from conversation history.
  - `record_interaction()` — persist turn data for model building.
- **Integration pattern**:
  - Session key mapping via `build_session_key(platform, user_id, thread_id)`.
  - Prefetch strategy: background thread loads user context during previous turn.
  - Injection at `before_model_request` or session-start.
  - Non-blocking: turn proceeds without user context if prefetch is delayed.
- **Distinction from content memory**:
  - Scope: user identity & preferences vs. task/content facts.
  - Persistence: external service (survives agent resets) vs. within agent storage.
  - Intelligence: dialectic Q&A (LLM-powered synthesis) vs. keyword/vector search.
  - Multi-agent: shared across agents serving the same user vs. per-agent.
