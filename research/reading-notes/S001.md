## 1. Core Claim
OpenClaw 的核心记忆哲学是**"文件即真相"（files-as-source-of-truth）**：代理的所有持久记忆都以纯 Markdown 文件形式保存在工作区磁盘上，LLM 本身不维护任何隐藏状态。模型"记住"的唯一方式是通过文件读写工具将信息写入磁盘。

## 2. Method / Mechanism
- **三层文件架构**：
  - `MEMORY.md`：精心策划的长期记忆，包含持久事实、偏好和决策。在每个主会话开始时注入系统提示。
  - `memory/YYYY-MM-DD.md`：每日工作笔记，记录运行上下文和观察。今天和昨天的笔记自动加载，但不注入每轮引导提示。
  - `DREAMS.md`（可选）：梦境扫描摘要，用于人类审阅的历史回填条目。
- **记忆蒸馏循环**：代理被期望定期将每日笔记中的有用材料提炼到 `MEMORY.md`，并删除过时条目。由生成的工作区指令和 heartbeat 流自动执行。
- **预算感知截断**：如果 `MEMORY.md` 超过 bootstrap 文件预算，磁盘上的文件保持完整，但注入模型上下文的副本会被截断。
- **工具暴露**：`memory_search`（语义搜索）和 `memory_get`（读取特定文件/行范围）。
- **memory-wiki 插件（可选）**：将持久知识编译为带 provenance 的 wiki vault，添加结构化声明、矛盾检测、新鲜度跟踪和 wiki 原生工具。

## 3. Key Evidence
- 官方文档明确声明："The model only 'remembers' what gets saved to disk — there is no hidden state."
- 三个记忆文件有明确的分工和加载策略（MEMORY.md 注入引导提示 vs daily notes 仅索引可搜索）。
- `/context list`、`/context detail`、`openclaw doctor` 命令可查看原始大小 vs 注入大小的截断状态。

## 4. Limitations & Gaps
- **无自动验证机制**：代理可能写入错误或矛盾的信息到 MEMORY.md，系统本身不验证内容准确性。
- **截断信号弱**：当 MEMORY.md 超过预算时，仅通过上下文大小报告提示用户，没有自动的分层降级策略。
- **依赖代理自觉性**：记忆蒸馏（从 daily notes 到 MEMORY.md）依赖代理主动执行，而非强制架构约束。
- **对比 MemGPT**：缺乏像 MemGPT 那样的操作系统级内存分页和进程隔离机制。

## 7. Personal Annotation
- OpenClaw 的"无隐藏状态"设计是一个强架构约束，但也意味着如果代理忘记写入文件，信息就永久丢失。这与人类记忆的"不可靠性"有趣地相似。
- memory-wiki 插件的出现表明 OpenClaw 团队已经意识到纯文件记忆的规模化和结构化问题，正在向"知识库"方向演进。
