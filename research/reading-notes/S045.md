## 1. Core Claim
- Doubao phone and AutoGLM represent two fundamentally different AI phone technical routes: Doubao uses system-level deep integration with direct GPU frame buffer reads and input event injection (high privilege, facing platform bans), while AutoGLM uses external ADB-based control through standard debugging interfaces (lower privilege, more open and compliant).

## 2. Method / Mechanism
**Doubao phone (system-level agent):**
- **Screen capture**: Directly reads GPU rendering buffer (frame buffer) instead of screenshot APIs; can bypass secure video output protections.
- **Input injection**: Uses `inject input event` with system-signed privileges, more direct than accessibility services.
- **Virtual screen**: Creates a headless virtual screen (brightness 0, resolution matches physical screen) for background task execution without interfering with foreground user activity.
- **Cloud-centric**: ~250KB uploads every 3–5s to cloud model; ~1KB action instructions returned.

**AutoGLM (external ADB agent):**
- **Architecture**: Agent runs on PC/cloud, controls phone via ADB or cloud phone APIs.
- **Screen capture**: Uses `screencap` via ADB or cloud phone stream—cannot bypass secure content protections.
- **Input execution**: `adb shell input tap/swipe` plus ADB Keyboard for text input.
- **Safety**: Phone Use framework with human-in-the-loop confirmation; cloud phone isolation reduces risk.

## 3. Key Evidence
- Doubao phone's virtual screen has **independent focus**—clicks on virtual screen don't affect physical screen.
- AutoGLM requires **developer mode + USB debugging**; Doubao phone uses system-signed components.
- Both routes rely on **cloud-based decision making**; local phone mainly captures and executes.
- Doubao phone's efficiency threatens **attention-economy business models** (7×24 automation replaces human attention).
